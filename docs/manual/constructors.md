    <article class="markdown-body entry-content container-lg" itemprop="text"><h1><a id="user-content-constructors" class="anchor" aria-hidden="true" href="#constructors"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Constructors</h1>
<p>کانستراکتورها توابعی هستند که اشیا جدید می‌سازند- به خصوص نمونه‌هایی از انواع مرکب. در جولیا اشیا نوع به عنوان توابع کانستراکتور هم هستند: آنها از خودشان نمونه‌های جدیدی می‌سازند هنگامیکه در یک آرگمان چندتایی به عنوان تابع استفاده می‌شوند. این نکته کم‌و بیش به طور مختصر پیش از این ذکر شد هنگامی که انواع مرکب معرفی شدند. برای مثال:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="julia&gt; struct Foo

<p>برای خیلی از انواع، درست کردن اشیا جدید با بایند کردن مقدار فیلدشان تنها چیزی است که نیاز است تا نمونه‌های جدید ساخته شوند. اما، در برخی موارد کارایی بیشتری هنگام درست کردن انواع مرکب نیاز است. بعضی اوقات اینورینت ها باید اعمال شوند، یا با چک کردن آرگومان‌ها یا با انتقال آن‌ها. داده‌ساختارهای بازگشتی، خصوصا آن‌هایی که احتمالا به خودشان رفرنس می‌دهند، عموما نمی‌توانند به خوبی درست بشوند مگر اینکه ابتدا به صورت ناقص باشند و سپس با تغییرات برنامه‌نویسی به کامل شوند که به عنوان یک قدم جداگانه از ساختن شی محسوب می‌شود. برخی اوقات، منطقی است تا بتوان اشیا را با تعداد کمتر پارامتر یا پارامتر‌های متفاوت به جای یک یا چند فیلد درست کرد. سیستم جولیا برای ساختن اشیا برای تمامی این موارد و بیشتر از این آماده شده است.</p>
<p>[^1]:
Nomenclature: while the term "constructor" generally refers to the entire function which constructs
objects of a type, it is common to abuse terminology slightly and refer to specific constructor
methods as "constructors". In such situations, it is generally clear from the context that the term
is used to mean "constructor method" rather than "constructor function", especially as it is often
used in the sense of singling out a particular method of the constructor from all of the others.</p>
<h2><a id="user-content-outer-constructor-methods" class="anchor" aria-hidden="true" href="#outer-constructor-methods"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Outer Constructor Methods</h2>
<p>کانستراکتورها دقیقا مانند دیگر توابع در جولیا هستند به این معنی که رفتار کلی آنها توسط رفتار ترکیبی متد‌هایشان تعریف می‌شوند. شما می‌توانید به سادگی با تعریف کردن متدهای جدید فانکشنالیتی به یک کانستراکتو اضافه کنید .
برای مثال فرض کنید می‌خواهیم به یک متد کانستراکتر برای شی foo اضافه کنیم که تنها یک آرگومان می‌گیرد و از مقدار داده‌شده برای هر دو فیلد baz‌و bar استفاده کند. این ساده‌است:
</p>

<p>اینجا صفر-آرگومان متد کانستراکتور تک-آرگومان متد کانستراکتور را فرا‌می‌خواند، که در مقابل به صورت خودکار متد کانستراکتور دو آرگومانه را فرا‌می‌خواند. به دلایلی که به زودی روشن خواهد شد متد کانستراکتور های اضافی که به عنوان متد‌های عادی فراخوانده می‌شوند به عنوان متد کانستراکتور بیرونی آن‌ها را خواهیم شناخت. این متد کانستراکتور های بیرونی تنها می‌توانند با صدا زدن یک کانستراکتور متد دیگر یک نمونه جدید درست کنند، مثل نمونه بالا که با مقادیر از پیش تعیین شده ۱ این‌کار را انجام داد.</p>
<h2><a id="user-content-inner-constructor-methods" class="anchor" aria-hidden="true" href="#inner-constructor-methods"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Inner Constructor Methods</h2>
<p>با اینکه متد کانستراکتور های بیرونی توانستد مشکل درست کردن متد های اضافی برای درست کردن اشیا را برطرف سازند؛ اما نمی‌توانند دو مشکل دیگر که در مقدمه این فصل ذکر شده را حل کنند: اعمال اینورینت‌ها و اجازه دادن ساختن اشیا خود رفرنس ده. برای حل این مشکلات نیاز به متد کانستراکتور های درونی داریم. یک متد کانستراکتور درونی دقیقا مثل یک متد کانستراکتور بیرونی است با این تفاوت که :</p>
<ol>
<li>１.	درون بلوک یک اعلام تایپ صدا زده می‌شود.( به جای آنکه در خارج آن صدا زده شود.)</li>
<li>２.	به یک تابع مخصوص که تنها به صورت محلی موجود است به نام new دسترسی دارد که می‌تواند اشیائی از نوع خود بلاک بسازد</li>
</ol>
<p>برای مثال فرض کنید می‌خواهیم یک نوع را که حاوی یک جفت عدد حقیقی است صدا بزنیم، اما این شرط اضافی را داریم که عدد اول از عدد دوم بزرگ‌تر نیست. شما می‌توانید این کار را به صورت زیر انجام دهید:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="julia&gt; struct OrderedPair

<p>اگر نوع صدا زده شده قابل تغییر بود، شما می‌توانید به طور مستقیم مقدار این‌گونه فیلد ها را تغییر دهید تا این اینورینت‌ها را نقض کنید. البته که دستکاری مقادیر ناخوانده درونی یک شی یک تمرین بد است!. شما همچنین می‌توانید متد های کانستراکتور بیرونی را در هرلحظه‌ای بعدا اضافه کنید، اما هنگامی که نوع صدا زده شد، دیگر نمی‌تواندی هیچ متد کانستراکتور درونی جدید اضافه کنید. از آنجایی که متد کانستراکتورهای بیرونی تنها می‌توانند با استفاده از فراخوانی دیگر کانستراکتورها اشیا جدید درست کنند، درنهایت اما برخی کانستراکتور های درونی باید برای ساخته شدن یک شی صدا زده شوند. این کار تضمین می‌کند که تمام اشیا از نوع صدا زده شده باید تنها با صدا زدن یک متد کانستراکتور درونی که با نوع مورد نظر تهیه شده است بوجود بیاید، درنتیجه تا حدی اعمال انورینت نوع خواهیم داشت.</p>
<p>اگر یک متد کانستراکتور درونی تعریف شده باشد، هیچ متد کانستراکتور از پیش‌تعیین شده‌ای تهیه نشده است: درواقع فرض شده که شما خودتان تمام متد‌های کانستراکتوری که نیاز داشته‌اید را تهیه کرده‌اید. کانستراکتور از پیش تعیین شده معادل نوشتن متد کانستراکتور داخلی خودتان است که تمام فیلدهای اشیا را به عنوان پارامتر( که باید به همان نوع درست خود فیلد باشند، اگر که فیلد نوع داشت.) می‌گیرید و آن‌هارا به new پاس می‌دهد و شی بازگردانده شده توسط آن را برمی‌گرداند.</p>

<p>تا جای ممکن کمترین متدهای کانستراکتور درونی یک تمرین خوب است: تنها آنهایی که تمام آرگومان ها را به طور صریح می‌گیرند و تبدیل و چک کردن خطای ضروری را اعمال می‌کنند. متدهای کانستراکتور اضافی، گذاشتن مقادیر از پیش تعییین شده یا تبدیل های کمکی باید توسط کانستراکتور بیرونی که کانستراکتور درونی را صدا می‌زند انجام شود. این تقسیم وظایف عموما طبیعی است.</p>
<h2><a id="user-content-incomplete-initialization" class="anchor" aria-hidden="true" href="#incomplete-initialization"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Incomplete Initialization</h2>
<p>مشکل آخری که هنوز حل نشده است اشیا خود رفرنس‌ده است یا به طور کلی داده ساختارهای بازگشتی. از آنجایی سختی اصلی این کار ممکن است واضح نباشد، بگذارید تا به طور خلاصه آن را توضیح دهیم. صدازدن بازگشتی نوع زیر را در نظر بگیرید:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="julia&gt; mutable struct SelfReferential

<p>این نوع ممکن است بی‌ضرر بنظر برسد اما هنوز چگونگی نمونه ساختن از آن را در نظر نگرفتیم. اگر a یک نمونه از SelfReferential باشد، آنگاه دومین نمونه می‌تواند به صورت زیر ساخته شود:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="julia&gt; b = SelfReferential(a)

<p>برای اینکه مقداردهی ناقص برای یک شی را اجازه دهیم جولیا اجازه می‌دهد تا تابع new با تعداد کمتری از تعداد کل فیلد‌هایی که خود شی دارد صدا زده شود، و یک شی با فیلد‌های مقدار دهی نشده برگرداند. پس از این می‌توانیم از متد کانستراکتورهای درونی برای این شی ناقص استفاده کنیم و مقداردهی آن را پیش از بازگرداندن آن کامل کنید. اینجا برای مثال یک تلاش دیگر برای تعریف نوع SelfReferential است. اینبار با استفاده از یک متد کانستراکتور  صفر-آرگومان ‌که فیلد obj شان به خودشان اشاره می‌کند این کار را کردیم. </p>

<h2><a id="user-content-parametric-constructors" class="anchor" aria-hidden="true" href="#parametric-constructors"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Parametric Constructors</h2>

<p>همانطور که می‌بینید برای صدا زدن کانستراکتور با بیان صریح نوع پارامترها، آرگومان‌ها به نوع فیلد‌های ضمنی تبدیل می‌شوند: Point{Int64}(1,2) کار می‌کند ولی Point{Int64}(1.0, 2.5) خطای InexactError را هنگام تبدیل 2.5 به Int64 درست می‌کند. هنگامی که نوع به طور ضمنی توسط آرگومان های کانستراکتور صدا زده تشکیل می‌شود، مثل Point(1,2)، باید نوع آرگومان‌ها با آنچه که باید یکسان باشند و گرنه T معلوم نخواهد شد. اما هر جفت از آرگومان های حقیقی با نوع موافق ممکن است به کانستراکتور  عمومی Point داده شود.</p>
<pPoint{Float64 و Point{Int64}  دو تابع کانستراکتور متفاوتند. در واقع Point{T} برای هر یک انواع ممکن برای T  یک کانستراکتور مجزاست. بدون ایجاد هیچ‌گونه کانستراکتور صریح فراخوانی نوع مرکب Point{T<:Real} به طور خودکار یک کانستراکتور درونی Point{T} برای هرکدازم از انواع T<:Real درست می‌کند که دقیقا مثل کانستراکتور درونی غیر پارامتری پیشفرض عمل می‌کند. این کانستراکتور همچنینی یک کانستراکتور کلی بیرونی هم درست می‌کند که جفت‌های آرگومان حقیقی میگیرد که باید از یک نوع باشند. این‌ ایجاد خودکار کانستراکتورها معادل فراخوانی صریح زیر است:</p>

<p>توجه کنیدکه هر تعریف شبیه نوع صدا زدن کانستراکتوری است که آن کار را انجام می‌دهد. صدا زدن Point{Int64}(1,2) تعریف Point{T}(x,y) داخل بلوک struct صدا می‌زند. فراخوانی کانستراکتور بیرونی اما یک متد کانستراکتور عمومی برای Point تعریف می‌کند که تنها هنگامی که جفت مقادیر از یک نوع حقیقی باشند به کار می‌آید. این فراخوانی صدازدن کانستراکتور را بدون داشتن صریح نوع پارامتر‌ها ممکن می‌سازد. مثل Point(1,2) یا Point(1.0,2.5). از آنجایی که فراخوانی متد روی آرگومان‌ها محدودیت یکسان بودن نوع را می‌گذارد، فراخوانی‌هایی همچون Point(1,2.5) که در آن آرگومان انواع متفاوتی دارند به خطای “no method”  منجر می‌شود.</p>
<p>فرض کنید می‌خواهیم صدا زدن‌هایی همچون Point(1,2.5) را با کمک “promoting” یا ارتقا 1 به 1.0 که ممیز شناور است ممکن سازیم. ساده‌ترین راه برای این‌کار اینست که کانستراکتور بیرونی اضافی زیر را تعریف کنیم:</p>

<p>در خط اول struct OurRational{T<:Integer} <: Real بیان می‌دارد که OurRational  یک پارامتر نوع integer می‌گیرد و خودش هم یک نوع real است. بیان فیلد‌های num::T و den::T نشان می‌دهد که داده ذخیره شده در شی OurRational{T} یک جفت از integerهای از نوع T هستند که یکی از آنها مقدار گویای صورت و دیگری مقدار مخرج را نشان می‌دهد.</p>
<pOurRational یک کانستراکتور درونی دارد که چک میکند که num و den هر دو همزان صفر نباشند و مطمئن می‌شود که کسر گویا در ساده‌ترین شکل ممکن خود با کوچکترین مقدار ممکن مثبت در مخرج است که توسط تابع ب.م.م یا همان gcd بالا محاسبه شده‌است. از آنجایی که gcd بزرگترین مقسوم‌علیه مشترک آرگومان‌هایش را  با حفظ ساین نخستین آرگومان که اینجا den است برمی‌گرداند، پس از تقسیم، مقدار جدید denتضمین می‌شود که نامنفی است. به این دلیل که این کانستراکتور تنها کانستراکتور درونی برای OurRational است، می‌توان مطمئن بود که اشیا OurRational همواره از طریق همین کانستراکتور و به همین شکل تولید خواهند شد.</p>
<p>OurRational همچنین یکسری کانستراکتور بیرونی هم برای اطمینان تهیه می‌کند. اولین کانستراکتور عمومی استاندارد که نوع T را از نوع صورت و مخرج که یکسانند استخراج می‌کند. دومین کانستراکتور هنگامی که صورت و مخرج از دو نوع متفاوتند به کار می‌آید: این کانستراکتور آن‌ها را به یک نوع مشترک ارتقا می‌دهد و سپس کار را به کانستراکتور قبلی پاس می‌دهد. سومین کانستراکتور بیرونی هم مقادیر صحیح را به نوع گویا با اضافه کردن یک به مخرج‌ آن‌ها می‌برد.</p>
<p>با دنبال کردن تعریف کانستراکتور های بیرونی ما یکسری متد برای عملگر ⊘  که امکان نوشتن اعداد گویا به صورت 1⊘2 را به ما می‌دهد. نوع Rational خود جولیا از عملگر // برای این هدف استفاده می‌کند.  پیش از این نماد ⊘ کاملا بدون معنا بوده، پس از اما ⊘ کاملا متفاوت عمل می‌کند و تمام رفتار آن در همین چند خط تعریف شده‌است. اولین و مهمترین تعریف کارایی این عملگر آن است که با a⊘b  یک OurRational دقیقا مشابه آنچه که کانستراکتور‌های آن هستند تولید کند. هنگامی که یکی از عملوند‌ها پیش از این گویاست، ما یک عدد گویا جدید برای نسبت منتج شده درست می‌کنیم که کمی با قبل متفاوت است: این عملکرد درواقع دقیقا همان عملکردی‌است که هنگام تقسیم یک عدد گویا به یک عدد صحیح مشاهده می‌کنیم. درنهایت، استفاده کردن از این عملگر هنگامی که عملوند‌ها مختلط با مقادیر صحیح هستند منجر به تولید یک نمونه از Complex{OurRational می‌شود که یک عدد مختلط است که قسمت حقیقی و موهومی آن گویا هستند.</p>
