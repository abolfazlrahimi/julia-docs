# ۱۴. تبدیل و ارتقا
<p>جولیا یک سیستم برای ارتقاء آرگومان های عملگر های ریاضیاتی به تایپ مشترک دارد، که در بسیاری از بخش های دیگر به آن اشاره شده است. بخش هایی مثل : اعداد صحیح و اعداد ممیزی شناور، عملگرهای ریاضیاتی و توابع اولیه، تایپ ها و روش ها. در این بخش، توضیح خواهیم داد که این ارتقاء سیستم چگونه کار میکند. همچنین نشان خواهیم داد که این ارتقاء را چگونه برای تایپ های جدید گسترش بدهیم و چگونه برای توابعی جز توابعی که با عملگرهای ریاضیاتی ساخته شده اند به کار گیریم. به طور معمول، زبان های برنامه نویسی با در نظر گرفتن ارتقاء آرگومان های حسابگر به دو بخش تبدیل میشوند</p>
<ul>
<li><strong>•	ارتقاء خودکار برای تایپ های حسابگر و عملگرها :</strong> در بسیاری از زبان ها، تایپ های عددی وقتی به عنوان عملوند هایی برای عملگر های حسابگر با ترکیبات پیوندی، مثل <code>+</code>,
<code>-</code>, <code>*</code>, و <code>/</code>,  استفاده میشوند، برای به دست آوردن جواب مورد انتظار به طورخورکار به تایپ معمول ارتقاء پیدا میکنند. زبان هایی مثل سی، جاوا، پرل و پایتون، به درستی جمع 1.5 + 1 را محاسبه کرده و به صورت 2.5 که یک مقدار ممیزی شناور است نشان می دهند. حتی با اینکه عملوند ها برای عملگر + اعداد صحیح هستند. این سیستم ها بسیار مناسب و راحت هستند و طوری دقیق طراحی شده اند که بیشتر برنامه نویسان موقع نوشتن این اصطلاحات اصلا متوجه این ارتقاء ها نمیشوند؛ اما کامپایلر ها و مفسران باید عملیات تبدیل را قبل از جمع زدن انجام دهند چرا که اعداد صحیح و ممیزی شناور نمیتوانند به همان صورتی که هستند اضافه شوند.</li>
<li><strong>•	ارتقاء های غیرخودکار</strong> این بخش “Ada” و “ML” را در بر دارد که زبان هایی از نوع آماری بسیار سخت هستند. در این زبان ها، هر تبدیل باید به طور صریح توسط برنامه نویس مشخص شود. برای مثال، مانند مثال قبل اصطلاح 1.5 + 1  در هر دو زبان یک اشتباه تالیفی انگاشته میشود. در عوض باید آن را به صورت “real(1) + 1.5” نوشت که صریحا عدد صحیح 1 را به یک عدد ممیزی شناور تبدیل کرده و سپس عملیات جمع را انجام میدهد. به کارگرفتن تبدیل صریح هرجای که نیاز است کار بسیار ناخوشایند نیست هرچند که حتی زبانی مثل “Ada” هم تا حدودی از تبدیل های خودکار استفاده میکند: اعداد صحیح لیترال به تایپ صحیح مورد انتظار ارتقاء پیدا میکنند و اعداد ممیزی شناور لیترال به تایپ ممیزی شناور مناسب ارتقاء پیدا میکنند.</li>
</ul>
<p>از یک جهت جولیا در دسته " ارتقاء غیرخودکار" قرار میگیرد: عملگر های ریاضیاتی صرفا توابعی با ترکیبات بخصوص هستند و آرگومان های توابع هیچ موقع به طور خودکار تبدیل نمیشوند. اگرچه، یکی ممکن است مشاهده کند که اعمال کردن عملگرهای ریاضیاتی به یک دسته متنوع از تایپ های ارگومان های مختلط یک مورد بسیار سخت از ارسال پولیمرفیک چندگانه است – چیزی که رسیدگی به آن در ارسال جولیا و تایپ سیستم ها به طور ویژه ای مناسب و آسان است. ارتقاء "خودکار" عملوند های ریاضیاتی به سادگی به عنوان یک کاربرد ویژه نمایان میشود : جولیا قوانین از پیش تعیین شده و همه گیر برای ارسال عملگر های ریاضیاتی دارد، وقتی که هیچ پیاده سازی بخصوصی برای بعضی از ترکیبات تایپ های عملوند وجود ندارد،فرا میخواند. این قوانین همه گیر ابتدا تمام عملوند ها را با استفاده از قوانین ارتقاء قابل تعریف توسط کاربر به تایپ معمول ارتقاء می دهد، سپس با استفاده از مقادیر به دست آمده به عنوان عملوند،که اکنون از یک نوع هستند،پاسخ را بدست می آورند.تایپ های قابل تعریف توسط کاربر میتوانند به سادگی با تعریف روش هایی برای تبدیل به و از فرم های دیگر در این سیستم ارتقاء شرکت کنند و فقط تعداد انگشت شماری از قوانین ارتقاء تعریف میکنند که چه تایپی را باید یه چه تایپی تغییر دهند وقتی با بقیه تایپ های میکس شده است.</p>
<h2><a id="user-content-conversion" class="anchor" aria-hidden="true" href="#conversion"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>۱۴.۱ تبدیل</h2>
<p>راه استاندارد بدست آوردن مقدار یک نوع خاص T اینست که کانستراکتور نوع‌آن را فرابخوانیم، T(x). اما مواردی وجود دارد که باید بتوانیم یک مقدار را به یک نوع دیگر تغییر دهیم بدون آنکه برنامه نویس از نوع آن به طور صریح بپرسد. یک مثال جایگذاری یک مقدار در یک آرایه است: اگر A یک Vector{Float64} باشد، کد A[1]=2 باید به صورت اتوماتیک با تبدیل 2 به یک ممیز شناور و سپس ذخیره کردن نتیجه در آرایه کار کند. این عملکرد با استفاده از تابع  convert صورت میگیرد

<p>برخی زبان ها رشته‌ها را مانند اعداد تفسیر می‌کنند یا برخی دیگر اعداد را با فرمت کردن به عنوان رشته درنظر می‌گیرند تا بتوانند تبدیلات را داشته باشند(بسیاری از زبان‌های پویا برای برنامه‌نویس به صورت خودکار تبدیلات را انجام می‌دهند.) جولیا خلاف این‌کار را انجام می‌دهد. با اینکه برخی رشته می‌توانند به عنوان اعداد تفسیر شوند، امام بیشتر رشته‌ها برای نمایش به صورت اعداد مناسب نیستند و تنها زیرمجموعه محدودی از آنها می‌توانند. به همین دلیل جولیا تابع parse را اختصاص داده است تا هنگام چنین اعمالی استفاده شود که در حالت کلی این کار را صریح می‌کند</p>
<h3><a id="user-content-when-is-convert-called" class="anchor" aria-hidden="true" href="#when-is-convert-called"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>چه زمانی convert صدا زده می‌شود؟</h3>
<p>کانستراکتور‌های زیر در زبان convert را صدا می‌زنند.</p>
<ul>
<li>•	مقداردهی یک آرایه تبدیل به المنت آرایه را انجام می‌دهد.</li>
<li>•	مقداردهی به یک فیلد یک شی تبدیل را به نوع آن فیلد تغییر می‌دهد.</li>
<li>•	ساختن یک شی با new تبدیل را به نوع فیلدهای فراخوانده شده انجام می‌دهد.</li>
<li>•	مقداردهی به یک متغیر که یک نوع اظهار شده دارد(برای مثال local x::T) تابع تبدیل به نوع گفته شده را صدا می‌زند.</li>
<li•	مقدار بازگردانده شده یک تابع که نوع بازگرداندن آن از قبل تعیین شده هم تابع تبدیل را فرا‌می‌خواند.</li>
<li>•	پاس دادن یک مقدار به ccall تبدیل به نوع آرگومان آن را انجام می‌دهد.</li>
</ul>
<h3><a id="user-content-conversion-vs-construction" class="anchor" aria-hidden="true" href="#conversion-vs-construction"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Conversion vs. Construction</h3>
<p>خوب است توجه کنیم که رفتار تابع convert(T,x) تقریبا با T(x) یکسان است. البته عموما این‌گونه‌ است. اما یک تفاوت معنایی وجود دارد: از آنجایی که convert می تواند به صورت ضمنی فراخوانده شود، متدهایش محدود به مواردی هستند که به عنوان “safe” یا “unsurprising” قرار می‌گیرند. Convert تنها میان نوع‌هایی تبدیل را انجام می‌دهد که یک نوع پایه یکسان یک چیز را نشان می‌دهند( برای مثال نمایش های گوناگون اعداد، یا encoding های متفاوت یک رشته). این کار همچنین عموما بدون تلفات است یعنی تبدیل یک مقدار به یک نوع متفاوت و سپس تبدیل دوباره آن باید به مقدار اولیه منتج شود.</p>
<p>۴ حالت عمومی متفاوت وجود دارد که در آن‌ها کانستراکتور از convert تفاوت دارد:</p>
<h4><a id="user-content-constructors-for-types-unrelated-to-their-arguments" class="anchor" aria-hidden="true" href="#constructors-for-types-unrelated-to-their-arguments"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>１.	کانستراکتور برای انواعی که نسبت به آرگومانشان بی ربطند.</h4>
<p>بعضی از کانسترکتورها مفهوم تبدیل یا conversion   را پیاده‌سازی نمی‌کنند. برای مثال، Timer(2) یک زمان‌بندی ۲ ثانیه‌ای درست می‌کند که به طور کلی یک تغییر از عدد صحیح به دامنه زمان نیست</p>

<h2><a id="user-content-promotion" class="anchor" aria-hidden="true" href="#promotion"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Promotion</h2>
<p>Promotion refers to converting values of mixed types to a single common type. Although it is not
strictly necessary, it is generally implied that the common type to which the values are converted
can faithfully represent all of the original values. In this sense, the term "promotion" is appropriate
since the values are converted to a "greater" type -- i.e. one which can represent all of the
input values in a single common type. It is important, however, not to confuse this with object-oriented
(structural) super-typing, or Julia's notion of abstract super-types: promotion has nothing to
do with the type hierarchy, and everything to do with converting between alternate representations.
For instance, although every <code>Int32</code> value can also be represented as a <code>Float64</code> value,
<code>Int32</code> is not a subtype of <code>Float64</code>.</p>
<p>Promotion to a common "greater" type is performed in Julia by the <code>promote</code> function, which takes
any number of arguments, and returns a tuple of the same number of values, converted to a common
type, or throws an exception if promotion is not possible. The most common use case for promotion
is to convert numeric arguments to a common type:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="julia&gt; promote(1, 2.5)

<p>Floating-point values are promoted to the largest of the floating-point argument types. Integer
values are promoted to the larger of either the native machine word size or the largest integer
argument type. Mixtures of integers and floating-point values are promoted to a floating-point
type big enough to hold all the values. Integers mixed with rationals are promoted to rationals.
Rationals mixed with floats are promoted to floats. Complex values mixed with real values are
promoted to the appropriate kind of complex value.</p>

the most typical "clever" application being the definition of catch-all methods for numeric operations
like the arithmetic operators <code>+</code>, <code>-</code>, <code>*</code> and <code>/</code>. Here are some of the catch-all method definitions

<h3><a id="user-content-defining-promotion-rules" class="anchor" aria-hidden="true" href="#defining-promotion-rules"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Defining Promotion Rules</h3>
<p>Although one could, in principle, define methods for the <code>promote</code> function directly, this would
require many redundant definitions for all possible permutations of argument types. Instead, the
behavior of <code>promote</code> is defined in terms of an auxiliary function called <code>promote_rule</code>, which
one can provide methods for. The <code>promote_rule</code> function takes a pair of type objects and returns
another type object, such that instances of the argument types will be promoted to the returned
type. Thus, by defining the rule:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="promote_rule(::Type{Float64}, ::Type{Float32}) = Float64
"><pre><span class="pl-en">promote_rule</span>(<span class="pl-k">::</span><span class="pl-c1">Type{Float64}</span>, <span class="pl-k">::</span><span class="pl-c1">Type{Float32}</span>) <span class="pl-k">=</span> Float64</pre></div>
<p>one declares that when 64-bit and 32-bit floating-point values are promoted together, they should
be promoted to 64-bit floating-point. The promotion type does not need to be one of the argument
types. For example, the following promotion rules both occur in Julia Base:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="promote_rule(::Type{BigInt}, ::Type{Float64}) = BigFloat
promote_rule(::Type{BigInt}, ::Type{Int8}) = BigInt
"><pre><span class="pl-en">promote_rule</span>(<span class="pl-k">::</span><span class="pl-c1">Type{BigInt}</span>, <span class="pl-k">::</span><span class="pl-c1">Type{Float64}</span>) <span class="pl-k">=</span> BigFloat
<span class="pl-en">promote_rule</span>(<span class="pl-k">::</span><span class="pl-c1">Type{BigInt}</span>, <span class="pl-k">::</span><span class="pl-c1">Type{Int8}</span>) <span class="pl-k">=</span> BigInt</pre></div>
<p>In the latter case, the result type is <code>BigInt</code> since <code>BigInt</code> is the only type
large enough to hold integers for arbitrary-precision integer arithmetic. Also note that
one does not need to define both <code>promote_rule(::Type{A}, ::Type{B})</code> and
<code>promote_rule(::Type{B}, ::Type{A})</code> -- the symmetry is implied by the way <code>promote_rule</code>
is used in the promotion process.</p>
<p>The <code>promote_rule</code> function is used as a building block to define a second function called <code>promote_type</code>,
which, given any number of type objects, returns the common type to which those values, as arguments
to <code>promote</code> should be promoted. Thus, if one wants to know, in absence of actual values, what
type a collection of values of certain types would promote to, one can use <code>promote_type</code>:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="julia&gt; promote_type(Int8, Int64)
Int64
"><pre>julia<span class="pl-k">&gt;</span> <span class="pl-c1">promote_type</span>(Int8, Int64)
Int64</pre></div>
<p>Internally, <code>promote_type</code> is used inside of <code>promote</code> to determine what type argument values
should be converted to for promotion. It can, however, be useful in its own right. The curious
reader can read the code in <a href="https://github.com/JuliaLang/julia/blob/master/base/promotion.jl"><code>promotion.jl</code></a>,
which defines the complete promotion mechanism in about 35 lines.</p>
<h3><a id="user-content-case-study-rational-promotions" class="anchor" aria-hidden="true" href="#case-study-rational-promotions"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Case Study: Rational Promotions</h3>
<p>Finally, we finish off our ongoing case study of Julia's rational number type, which makes relatively
sophisticated use of the promotion mechanism with the following promotion rules:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="promote_rule(::Type{Rational{T}}, ::Type{S}) where {T&lt;:Integer,S&lt;:Integer} = Rational{promote_type(T,S)}
promote_rule(::Type{Rational{T}}, ::Type{Rational{S}}) where {T&lt;:Integer,S&lt;:Integer} = Rational{promote_type(T,S)}
promote_rule(::Type{Rational{T}}, ::Type{S}) where {T&lt;:Integer,S&lt;:AbstractFloat} = promote_type(T,S)
"><pre><span class="pl-en">promote_rule</span>(<span class="pl-k">::</span><span class="pl-c1">Type{Rational{T}}</span>, <span class="pl-k">::</span><span class="pl-c1">Type{S}</span>) <span class="pl-k">where</span> {T<span class="pl-k">&lt;:</span><span class="pl-c1">Integer</span>,S<span class="pl-k">&lt;:</span><span class="pl-c1">Integer</span>} <span class="pl-k">=</span> Rational{<span class="pl-c1">promote_type</span>(T,S)}
<span class="pl-en">promote_rule</span>(<span class="pl-k">::</span><span class="pl-c1">Type{Rational{T}}</span>, <span class="pl-k">::</span><span class="pl-c1">Type{Rational{S}}</span>) <span class="pl-k">where</span> {T<span class="pl-k">&lt;:</span><span class="pl-c1">Integer</span>,S<span class="pl-k">&lt;:</span><span class="pl-c1">Integer</span>} <span class="pl-k">=</span> Rational{<span class="pl-c1">promote_type</span>(T,S)}
<span class="pl-en">promote_rule</span>(<span class="pl-k">::</span><span class="pl-c1">Type{Rational{T}}</span>, <span class="pl-k">::</span><span class="pl-c1">Type{S}</span>) <span class="pl-k">where</span> {T<span class="pl-k">&lt;:</span><span class="pl-c1">Integer</span>,S<span class="pl-k">&lt;:</span><span class="pl-c1">AbstractFloat</span>} <span class="pl-k">=</span> <span class="pl-c1">promote_type</span>(T,S)</pre></div>
<p>The first rule says that promoting a rational number with any other integer type promotes to a
rational type whose numerator/denominator type is the result of promotion of its numerator/denominator
type with the other integer type. The second rule applies the same logic to two different types
of rational numbers, resulting in a rational of the promotion of their respective numerator/denominator
types. The third and final rule dictates that promoting a rational with a float results in the
same type as promoting the numerator/denominator type with the float.</p>
<p>This small handful of promotion rules, together with the type's constructors and the default
<code>convert</code> method for numbers, are sufficient to make rational numbers interoperate completely
naturally with all of Julia's other numeric types -- integers, floating-point numbers, and complex
numbers. By providing appropriate conversion methods and promotion rules in the same manner, any
user-defined numeric type can interoperate just as naturally with Julia's predefined numerics.</p>
